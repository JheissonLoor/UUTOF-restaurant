§ 6 · Modelo lógico & físico
006 / 012
Capítulo 6

DDL en MySQL.

Traducción directa del modelo conceptual a SQL. Se respetan claves foráneas, tipos enumerados y restricciones de integridad referencial con ON DELETE explícito.

Esquema de dominioMySQL 8 · InnoDB
-- Charset UTF-8 + InnoDB para FK y transacciones
CREATE DATABASE uttof
  CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
USE uttof;

-- Usuarios (clientes y empleados)
CREATE TABLE usuario (
  id_usuario     INT AUTO_INCREMENT PRIMARY KEY,
  nombre         VARCHAR(120) NOT NULL,
  email          VARCHAR(180) NOT NULL UNIQUE,
  password_hash  CHAR(60) NOT NULL,         -- bcrypt
  telefono       VARCHAR(20),
  rol            ENUM('cliente','mesero','cocina','admin')
                 NOT NULL DEFAULT 'cliente',
  creado_en      DATETIME NOT NULL
                 DEFAULT CURRENT_TIMESTAMP
) ENGINE=InnoDB;

-- Catálogo
CREATE TABLE categoria (
  id_categoria INT AUTO_INCREMENT PRIMARY KEY,
  nombre       VARCHAR(80) NOT NULL,
  orden        SMALLINT NOT NULL DEFAULT 0
) ENGINE=InnoDB;

CREATE TABLE platillo (
  id_platillo  INT AUTO_INCREMENT PRIMARY KEY,
  id_categoria INT,
  nombre       VARCHAR(120) NOT NULL,
  precio       DECIMAL(10,2) NOT NULL CHECK(precio >= 0),
  disponible   BOOLEAN NOT NULL DEFAULT TRUE,
  imagen_url   TEXT,
  FOREIGN KEY (id_categoria) REFERENCES categoria(id_categoria)
    ON DELETE RESTRICT
) ENGINE=InnoDB;
OperaciónMySQL 8 · InnoDB
-- Mesas y reservas
CREATE TABLE mesa (
  id_mesa   INT AUTO_INCREMENT PRIMARY KEY,
  numero    INT NOT NULL UNIQUE,
  capacidad SMALLINT NOT NULL,
  estado    ENUM('libre','ocupada','reservada','limpieza') NOT NULL DEFAULT 'libre'
) ENGINE=InnoDB;

CREATE TABLE reserva (
  id_reserva       INT AUTO_INCREMENT PRIMARY KEY,
  id_usuario       INT, id_mesa INT,
  hora_reserva     DATETIME NOT NULL,
  num_personas     SMALLINT NOT NULL CHECK(num_personas > 0),
  notas_especiales TEXT,
  FOREIGN KEY (id_usuario) REFERENCES usuario(id_usuario) ON DELETE SET NULL,
  FOREIGN KEY (id_mesa) REFERENCES mesa(id_mesa)
) ENGINE=InnoDB;

-- Pedido + detalle
CREATE TABLE pedido (
  id_pedido  INT AUTO_INCREMENT PRIMARY KEY,
  id_usuario INT, id_mesa INT,
  estado     ENUM('creado','en_cocina','listo','entregado','pagado','cancelado') NOT NULL DEFAULT 'creado',
  total      DECIMAL(10,2) NOT NULL DEFAULT 0,
  creado_en  DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  INDEX idx_pedido_estado (estado),
  FOREIGN KEY (id_usuario) REFERENCES usuario(id_usuario),
  FOREIGN KEY (id_mesa) REFERENCES mesa(id_mesa)
) ENGINE=InnoDB;

CREATE TABLE detalle_pedido (
  id_detalle  INT AUTO_INCREMENT PRIMARY KEY,
  id_pedido   INT NOT NULL, id_platillo INT NOT NULL,
  cantidad    SMALLINT NOT NULL CHECK(cantidad > 0),
  subtotal    DECIMAL(10,2) NOT NULL, notas TEXT,
  FOREIGN KEY (id_pedido) REFERENCES pedido(id_pedido) ON DELETE CASCADE,
  FOREIGN KEY (id_platillo) REFERENCES platillo(id_platillo)
) ENGINE=InnoDB;

-- Pagos
CREATE TABLE pago (
  id_transaccion INT AUTO_INCREMENT PRIMARY KEY,
  id_pedido      INT NOT NULL UNIQUE,
  metodo         ENUM('efectivo','tarjeta','yape','mixto') NOT NULL,
  monto          DECIMAL(10,2) NOT NULL,
  propina        DECIMAL(10,2) DEFAULT 0,
  fecha          DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (id_pedido) REFERENCES pedido(id_pedido)
) ENGINE=InnoDB;

Ciclo de vida del pedido.

Modelo postpago a la mesa: el pedido va directo a cocina sin paso intermedio de confirmación. El mesero marca cuando entrega el plato, y el cobro puede iniciarlo el cliente desde su app o el mesero desde su tablet.

Decisiones clave
· Sin estado confirmado — el pedido va directo a cocina al crearse.
· El mesero marca entregado al servir el plato en la mesa.
· Postpago: el cobro sucede después de comer; lo inicia cliente o mesero.
· Cada cambio escribe en audit_log con actor + timestamp.

Capítulo 11

Seguridad & RBAC.

Defensa en profundidad: TLS en transporte, hashing fuerte en reposo, JWT firmado para sesión y control de acceso por roles en cada endpoint.

Flujo de autenticación & autorización

Autenticación

JWT · OAuth-like
  • Access token HS256, vigencia 15 min, claims sub, rol, exp, jti.
  • Refresh token opaco, almacenado en HttpOnly Secure cookie.
  • OTP de 6 dígitos al registrarse, vigencia 5 min.
  • Rotación de tokens en cada refresh + lista negra de jti.

Autorización

RBAC · 4 roles
  • cliente — su propio perfil, pedidos y reseñas.
  • mesero — mesas, pedidos en piso, cobros.
  • cocina — KDS, transiciones a listo.
  • admin — todo + reportes, empleados, menú, reservas.
  • Decoradores @requires(rol) en cada handler.

Datos en reposo

Cifrado · hashing
  • Contraseñas con bcrypt cost 12 (≈ 250 ms / hash).
  • Sal por usuario incluida en el hash de 60 caracteres.
  • Datos sensibles (teléfono, dirección) cifrados a nivel columna con AES_ENCRYPT nativo de MySQL.
  • Respaldos cifrados AES-256, retención 30 días.

Transporte & superficie

TLS · OWASP
  • HTTPS forzado · HSTS 1 año · TLS 1.2+.
  • Validación de entrada con esquemas; protección contra inyección SQL vía consultas parametrizadas.
  • Rate limiting 60 req/min por IP, 1000/h por token.
  • Logs sin PII; trazabilidad por request_id.
